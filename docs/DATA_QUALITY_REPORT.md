## Temporal and new-account holdouts

The first **70%** of the
observed time span is training data. Training ends at step
**520**; evaluation begins at step
**521**.

| Artifact | Steps | Rows | Fraud rows | Fraud rate | Size |
| --- | --- | ---: | ---: | ---: | ---: |
| `future_time_holdout` | 521–743 | 280,613 | 2,432 | 0.8667% | 8.17 MiB |
| `new_account_holdout` | 521–743 | 279,847 | 2,427 | 0.8673% | 8.15 MiB |
| `train` | 1–520 | 6,082,007 | 5,781 | 0.0951% | 175.26 MiB |

`new_account_holdout` is the future-time subset whose origin account was never
seen at or before the training cutoff. It overlaps `future_time_holdout` by
design but never overlaps training. This makes overall future evaluation and
cold-start evaluation directly comparable. In this dataset,
**99.7270%**
of future rows have an unseen origin account, so the known-account future slice
is too small to treat as the only primary test.

## Leakage assessment

| Field | v0.2 modelling policy | Reason |
| --- | --- | --- |
| `isFraud` | Target only | Ground-truth label. |
| `isFlaggedFraud` | Exclude; rules baseline only | Existing policy output derived from transaction conditions. |
| `oldbalanceOrg`, `oldbalanceDest` | Exclude from PaySim models | Dataset publisher warns the balance state reflects cancelled detected fraud. |
| `newbalanceOrig`, `newbalanceDest` | Prohibit | Post-transaction state and directly leakage-prone. |
| `nameOrig`, `nameDest` | Grouping only | Raw high-cardinality IDs invite entity memorisation; history must be point-in-time. |
| `step` | Splitting/time derivation only | Raw step can proxy simulation regimes and campaigns. |
| `type`, `amount` | Allowed | Known from the transaction request at scoring time. |

The canonical point-in-time table is maintained in
[`DATA_DICTIONARY.md`](DATA_DICTIONARY.md).

## Limitations and release decision

1. PaySim is synthetic and represents one simulated month, not live customer
   behaviour or operational label delay.
2. The fraud injection mechanism and cancellation behaviour can create
   unrealistically easy signals.
3. Account identifiers are synthetic; the new-account rate is not evidence of
   a real deployment's cold-start rate.
4. `new_account_holdout` currently means unseen **origin** account. Destination
   novelty can be reported as a separate slice in a later experiment.
5. The v0.1 data stage is accepted only for leakage-controlled baseline work;
   it does not establish model quality.
